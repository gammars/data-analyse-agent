# Data Analyse Agent 保研简历项目整理稿

## 一、推荐放入简历版本（STAR 强化版）

2026.06-2026.07 Data Analyse Agent 本地结构化数据分析智能体 个人项目

项目背景：在本地多表 CSV/Excel 数据分析场景中，通用大模型容易出现三类问题：不了解数据表关系、把无关问题也强行转成 SQL、复杂统计分析能力受 SQL 表达能力限制；同时，直接执行模型生成的 SQL/Python 代码存在安全风险。

项目目标：设计并实现一套安全可控、可解释、可评测的本地结构化数据分析 Agent，使用户上传数据集后，能够通过自然语言完成查询、图表、数据质量分析和复杂统计分析，并能清楚看到 Agent 的规划过程、工具调用过程和分析依据。

核心行动：

> 数据建模与安全查询：基于 FastAPI、SQLite 和 Pandas 搭建本地数据分析后端，将 CSV/Excel 物化为 SQLite 数据集，维护 Manifest、表结构、字段信息和表间关系；实现 SQLService 只读执行层，结合 SQLite 只读 URI、`PRAGMA query_only=ON`、SQL authorizer、单语句校验、危险语句拦截、超时和最大行数限制，降低自然语言生成 SQL 的误操作风险。

> Agent 规划与工具路由：设计 ScopeRouter，在进入执行前判断用户问题属于 `in_scope/out_of_scope/needs_clarification/general_help`，避免天气、股市预测等无关问题误触发数据库分析；实现结构化 Planner，最多拆解 5 个执行步骤，并按 `query/chart/quality/advanced/cleaning/mixed` 为每步动态限制可见工具和重试次数，缓解“所有问题都走 SQL”的工具滥用问题。

> Python 沙箱复杂分析：实现 `python_analysis` 工具和 PythonSandboxService，将 SQL 查询结果导出为 JSON，再把生成的 Python 脚本与数据挂载进 Docker 容器执行，读取 `result.json` 和图表产物；加入脚本静态检查、网络禁用、CPU/内存/超时限制、输出截断和沙箱健康检查，使系统能够处理相关性分析、异常值检测、聚类、时间序列趋势和复杂可视化等 SQL 难以完成的任务。

> 可解释前端与评测闭环：前端展示执行计划、步骤状态、工具耗时、Python/SQL 代码高亮、图表产物和错误信息；构建 `evals/` 自动化评测流水线，覆盖 ScopeRouter、Planner、Tool Routing、SQL correctness/security 和 Python analysis/sandbox 等关键能力，完整模式 31/31 通过，项目单元测试 58 项通过。

项目结果：形成了“数据集建模 -> 范围识别 -> 结构化规划 -> 工具白名单执行 -> Artifact 归档 -> 最终解释”的闭环，使系统从简单问答工具升级为可调试、可扩展、可评测的数据分析 Agent 原型。

创新点与科研潜力：

> 受控工具可见性：不是把所有工具一次性交给 Agent，而是在 ScopeRouter 和 Planner 的基础上进行步骤级工具白名单控制，可作为研究“LLM Agent 工具选择可靠性”和“意图约束下的工具调用策略”的实验原型。

> SQL 与 Python 分层分析：将确定性数据库查询和开放式 Python 复杂分析拆分为两层，简单聚合走 SQL，复杂统计走沙箱 Python，为研究“结构化数据问答中的混合执行计划生成”提供了清晰框架。

> 可评测 Agent 工作流：围绕范围识别、计划生成、工具路由、SQL 安全和 Python 沙箱分别构建评测用例，使系统不仅能运行，还能被持续评估和迭代，具备进一步扩展为科研实验平台的潜力。

> 安全执行机制：从 SQL authorizer、只读连接到 Docker 沙箱、静态检查和资源限制，形成多层防护思路，可继续探索“LLM 生成代码的安全执行与自动修复”方向。

我做的是一个本地结构化数据分析智能体，目标是让用户上传 CSV 或 Excel 后，可以用自然语言完成数据查询、质量分析、图表生成和复杂统计分析。系统底层会把数据物化为 SQLite，并保存数据集的 Manifest、表结构、字段信息和关系配置。对于简单聚合类问题，Agent 通过受控 SQL 查询完成；对于相关性、异常值、聚类、时间序列趋势和复杂图表这类 SQL 不擅长的问题，系统会把查询结果导出成 JSON，再把生成的 Python 脚本放进 Docker 沙箱执行，最后读取 `result.json` 和图表产物交给 Agent 总结。

如果老师追问这个项目的课程相关性，可以这样说：

这个项目和数据库系统的结合点主要有三个：一是把半结构化文件规范化为可查询的 SQLite 数据集，并维护 Schema、索引候选和表间关系；二是围绕自然语言生成 SQL 做只读执行、权限控制、语句校验和结果限制；三是把数据库查询结果作为下游分析任务的数据源，形成数据库查询、Agent 规划和 Python 复杂分析之间的协同流程。

## 四、面向老师关注点的创新性表述

如果老师更关心创新点，可以这样组织：

我的项目不是简单调用大模型做数据问答，而是围绕“LLM 如何安全、可靠地使用数据库和分析工具”做了一个可运行原型。它的创新性主要体现在三点。

第一是工具调用的可控性。传统 Agent 往往把所有工具一次性暴露给模型，模型容易误选工具。我在系统里加入 ScopeRouter 和结构化 Planner，先判断问题是否属于当前数据集，再按步骤限制可见工具。这个设计把工具选择从“完全自由生成”变成“意图约束下的受控选择”，后续可以进一步研究不同工具可见性策略对 Agent 正确率和安全性的影响。

第二是 SQL 与 Python 的混合执行。SQL 适合确定性查询，但不适合复杂统计、聚类、异常检测和多步骤可视化；Python 分析能力强，但执行风险更高。我的系统把两者分层：SQL 负责获取受控数据子集，Python 沙箱负责复杂分析。这可以扩展成“结构化数据分析任务的自动执行计划生成”问题，例如研究什么时候应该走 SQL、什么时候应该走 Python、什么时候需要多步组合。

第三是可评测性。很多 Agent 项目停留在演示层面，很难判断改动后是否真的变好。我把系统拆成 ScopeRouter、Planner、Tool Routing、SQL 安全和 Python 沙箱几个可评测模块，并建立自动化评测流水线。后续可以继续扩展真实数据集和任务模板，用实验方式比较不同提示词、不同规划策略、不同工具约束策略的效果。

如果老师问“这个项目的科研潜力在哪里”，可以回答：

它可以继续往三个方向做研究：一是面向结构化数据的 Agent 工具选择可靠性评测；二是自然语言问题到 SQL/Python 混合执行计划的自动生成；三是大模型生成代码的沙箱执行、安全检测和失败自修复机制。这些方向都能在现有系统上继续加实验、加指标、加对比方法，而不是停留在应用开发。

## 五、可能被追问的问题与答题要点

1. 为什么不能所有问题都直接让大模型写 SQL？

参考回答：SQL 适合筛选、聚合、排序、分组等确定性查询，但相关性分析、异常值检测、聚类、时间序列趋势和多步骤图表往往需要 Pandas、NumPy、SciPy 或 Matplotlib 这类分析库。项目中采用分层策略，简单聚合走 `query_data`，基础图表走 `generate_chart`，复杂统计走 `python_analysis`，既保留 SQL 的确定性，又增强复杂分析能力。

2. Docker 沙箱解决了什么问题？

参考回答：大模型生成的 Python 代码存在误读文件、执行危险调用、消耗过多资源或污染宿主环境的风险。项目把脚本和数据挂载进容器，容器禁网并限制 CPU、内存和超时时间，同时只读取约定的 `/workspace/input/data.json` 并输出 `/workspace/output/result.json`，这样能把复杂分析能力和宿主环境隔离开。

3. ScopeRouter 和 Planner 有什么区别？

参考回答：ScopeRouter 是进入规划前的范围识别，用于判断问题是否和当前数据集相关、是否需要澄清或只是系统使用帮助。Planner 是在问题确认相关后，把任务拆成结构化步骤，并给每一步分配意图、工具白名单和重试次数。前者决定是否进入工具执行，后者决定如何执行。

4. 项目里最像数据库课程设计的部分是什么？

参考回答：包括文件数据集物化为 SQLite、Schema/Manifest 管理、表关系建模、只读 SQL 执行、安全校验、索引与主外键候选识别、多表查询支持，以及围绕数据库查询结果构建上层分析流程。这些都和数据库建模、查询执行、安全控制和数据管理有关。

5. 这个项目目前的局限是什么？

参考回答：第一，评测集规模还比较小，当前更偏第一版回归测试，后续需要扩充真实业务数据和更多问题模板；第二，Python 静态安全检查不能证明绝对安全，只能作为 Docker 隔离之外的补充；第三，当前执行计划是线性步骤，还没有实现 DAG 并行、多 Agent 协作和自动改计划；第四，对于超大数据集还需要进一步做采样、缓存、分块计算或列式存储优化。

6. 这个项目和普通数据分析平台相比，区别在哪里？

参考回答：普通数据分析平台更多依赖用户手动写 SQL、拖拽图表或编写脚本，而这个项目把自然语言、数据库查询、工具规划和 Python 复杂分析串成一个 Agent 工作流。它的重点不是单个分析函数，而是如何让模型在安全约束下选择正确工具、执行正确步骤，并把中间结果组织成可解释的分析链路。

7. 你怎么证明这个系统不是只做了一个 Demo？

参考回答：我没有只做前端演示，而是补了工程化闭环。第一，后端有 SQLService、PythonSandboxService、ScopeRouter、Planner、Artifact Store 等相对清晰的模块划分；第二，关键风险点都写了测试，包括 SQL 非 SELECT 拒绝、沙箱超时失败、工具注册、范围识别和规划策略；第三，专门建立了 `evals/` 自动化评测流水线，覆盖 Agent 执行过程中的多个阶段，便于后续迭代时做回归验证。
